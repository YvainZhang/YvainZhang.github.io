USB Wi-Fi 性能上不去，很多时候不是某个点单独慢，而是整条链路都不够顺。  
无线侧能力可能没什么问题，但主机这边一旦在中断、线程、聚合、回调和协议栈对接上没理顺，吞吐很容易就掉下来。

## 1. 性能瓶颈通常落在哪几条主线

实际看 USB Wi-Fi 驱动性能，通常先盯三件事：

- TX / RX 路径是否顺畅
- 上下文切换是否过多
- 多核分工是否合理

这三件事没顺之前，盯着某个函数抠几微秒，通常收效不大。

## 2. 典型发送路径

发送路径大体上就是：

1. 协议栈给出待发送数据
2. 驱动补充头部、描述符和控制信息
3. 数据进入发送队列
4. 驱动线程完成聚合与 USB 提交
5. 完成回调回收资源并上报状态

这条链上比较常见的问题有：

- 队列堵塞
- 聚合开销过高
- URB 提交粒度不合理
- 回收速度赶不上生产速度

## 3. 典型接收路径

接收路径通常是：

1. USB 完成中断到来
2. 回调拿到数据并写入接收环
3. 后续线程或软中断做解析
4. 必要时解聚合
5. 上送协议栈
6. 回填接收缓冲

接收吞吐上不去时，瓶颈经常不在空口，而在主机侧这些地方：

- 回调太碎
- 解聚合太重
- SKB 管理成本太高
- 回填不及时

## 4. 为什么 USB Wi-Fi 容易被主机侧拖住

因为它同时要处理：

- USB 中断
- 传输提交和完成
- 数据聚合 / 解聚合
- SKB 分配和释放
- 协议栈对接

这些环节里只要有一两个地方切换太碎，CPU 开销就会上得很快。

## 5. 真正有效的优化方向

### 资源池化

高频路径里最怕频繁动态申请 / 释放。  
因此常见优化包括：

- 预分配 URB ring
- 复用传输控制块
- 减少热路径上的 `kmalloc/kfree`

### 线程分工

如果中断处理、聚合、协议栈入口都挤在同一颗 CPU 上，竞争会非常明显。  
更合理的方式是让它们形成流水线，而不是混在一起抢时间片。

### 中断与协议栈协同

看性能时不要只盯驱动线程，最好把下面这些一起摆出来：

- USB IRQ 分布
- NET_RX / RPS 路径
- 驱动线程绑核
- 协议栈入口处理

因为最后真卡住的，往往不是单点绝对性能不够，而是整条吞吐链分布不均。

## 6. 多核调优真正想达到什么

多核调优也不是简单均摊工作量，更多是在压下面这些额外成本：

- cache 抖动
- 线程乒乓
- 中断和重处理互相抢占

理想一点的状态更像一条流水线：

- 一组核心更靠近硬件中断和回调
- 一组核心更靠近协议栈和控制路径

## 7. 排查时值得先看的指标

做 USB Wi-Fi 性能分析时，至少建议一起观察：

- 最终吞吐
- 各 CPU 占用
- 中断分布
- 聚合效率
- 解聚合成本
- 回收与回填是否滞后

只看最终吞吐的话，常常分不清问题到底在 USB、驱动线程、协议栈，还是 CPU 分工本身。

## 8. 总结

回到最后，USB Wi-Fi 驱动调优不太像单点提速，更像是在把整条链路梳顺：

- 中断顺
- 提交顺
- 聚合顺
- 回调顺
- 协议栈上送顺

最后真正能落地的调优思路，通常还是围绕三件事：

1. 热路径是否足够池化
2. 上下文切换是否过多
3. 多核是否形成了合理流水线
