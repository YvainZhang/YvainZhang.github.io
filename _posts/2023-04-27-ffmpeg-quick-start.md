很多人用 FFmpeg 的方式都差不多：先搜一条命令，跑通了就先用着。短期当然没问题，但概念一旦没理顺，后面就很容易踩坑：

- 明明只是换容器，却做了重编码
- 明明只想提音频，却把原始编码也改了

所以这篇不准备堆命令大全，而是先把 FFmpeg 在做什么讲明白，再看那些真正高频的场景。

## 1. FFmpeg 到底是什么

FFmpeg 不只是一个命令，而是一整套音视频工具链。最常见的几个名字分别负责不同事情：

- `ffmpeg`：做转码、抽流、合并、过滤、推流
- `ffprobe`：查看媒体信息
- `ffplay`：简单播放

它重要的原因也很直接：你日常能遇到的大部分媒体处理活，FFmpeg 基本都能接住：

- 容器转换
- 音视频转码
- 裁剪和拼接
- 抽帧和抽音频
- HLS / TS 等流媒体处理
- 批处理和脚本化处理

如果你做嵌入式多媒体、播放器、音频处理、视频平台，FFmpeg 都是很难绕开的基础工具。

## 2. 先把最容易混淆的几个概念拆开

### 容器不是编码

MP4、MKV、AVI、TS、WAV 这些首先是容器。容器决定的是“文件怎么装”，而不是“内容怎么压缩”。

比如一个 `mp4` 文件里，视频可能是 H.264，也可能是 H.265；音频可能是 AAC，也可能是 MP3。  
所以：

- **容器**是外壳
- **编码格式**是里面流的压缩方式

### 一个文件里可能不止一条流

容器内部通常有多条流，例如：

- 视频流
- 音频流
- 字幕流
- 附件流
- 章节信息或元数据

所以你会经常看到“抽音频”“抽字幕”“去视频流”这种说法，因为 FFmpeg 干的事，本来就是重新组织流，或者重新编码流。

### 复用和解复用

- 复用（mux）：把多条流按某个容器规则重新打包
- 解复用（demux）：把流从容器里拆出来

把这两个词先吃透，很多命令就不再像黑话了。例如：

- MP4 转 MKV，但音视频编码都不变，这其实只是重新复用
- 从视频里提取 AAC，这其实就是把音频流拆出来

### 编码格式和编码器不是一回事

H.264、H.265、AAC、MP3 是编码格式。  
`libx264`、`libx265`、`aac`、`libmp3lame` 是具体编码器实现。

同一种编码格式可能有多个编码器实现，它们在速度、压缩率、质量和许可证上都可能不同。

## 3. 使用 FFmpeg 前，先学会看输入

很多误操作都源于“还没看清输入文件就开始转”。最先应该养成的习惯是：

```bash
ffprobe input.mp4
```

或者：

```bash
ffmpeg -i input.mp4
```

你至少要先看清这几个信息：

- 容器类型
- 视频编码格式
- 音频编码格式
- 分辨率
- 帧率
- 采样率
- 声道数
- 码率

先把输入看清楚，后面这些判断才站得住：

- 是换容器就够了
- 还是必须转码
- 是否需要重采样
- 是否需要改声道数

## 4. 命令行结构应该怎么读

FFmpeg 命令看起来长，但结构其实很固定：

```bash
ffmpeg [全局参数] [输入相关参数] -i input [输出相关参数] output
```

几个最常用参数可以先记住：

- `-i`：指定输入
- `-y`：覆盖输出文件
- `-c:v`：指定视频编码器
- `-c:a`：指定音频编码器
- `-c copy`：直接复制流，不重编码
- `-vn`：不要视频
- `-an`：不要音频
- `-ss`：从某个时间点开始
- `-t`：持续多长时间
- `-ar`：音频采样率
- `-ac`：音频声道数
- `-b:a`：音频码率

其中最关键的是 `-c copy`。看到它就要条件反射地想到：这次大概率不是转码，而是在搬流。

## 5. 容器转换和转码，必须严格区分

这是 FFmpeg 初学者最容易搞混的点。

### 只换容器

```bash
ffmpeg -i input.mp4 -c copy output.mkv
```

这条命令做的事是：

- 读取 input.mp4 里的流
- 不改视频编码
- 不改音频编码
- 只是把它们重新装进 MKV 容器

优点是：

- 快
- 不损失质量
- CPU 消耗低

### 真正的转码

```bash
ffmpeg -i input.mp4 -c:v libx265 -c:a aac output.mp4
```

这时就不是“换外壳”，而是把视频和音频都重新编码了。它适合目标平台不支持原编码、或者你需要压缩体积、调整码率的情况。

因此判断一条命令是否重编码，只要看一眼：

- 有没有 `-c copy`
- 有没有显式指定新的编码器

## 6. 最常见的几个处理任务

### 1. 查看文件信息

```bash
ffprobe input.mp4
```

这是起点，不建议跳过。

### 2. 只换容器

```bash
ffmpeg -i input.mp4 -c copy output.mkv
```

适合“内容不用变，只是目标平台更喜欢另一个容器”的场景。

### 3. 提取音频

```bash
ffmpeg -i input.mp4 -vn -c:a copy output.aac
```

这里 `-vn` 表示丢掉视频流。  
如果原始音频流本来就是 AAC，这条命令会非常高效。

### 4. WAV 转 MP3

```bash
ffmpeg -i input.wav -c:a libmp3lame -b:a 192k output.mp3
```

这类命令适合有明确目标格式和码率要求的场景。

### 5. WAV 转 AAC

```bash
ffmpeg -i input.wav -c:a aac -b:a 128k output.m4a
```

如果目标平台偏向移动端或流媒体，AAC 通常比 MP3 更常见。

### 6. 截取一段音视频

```bash
ffmpeg -ss 00:00:10 -i input.mp4 -t 00:00:20 -c copy output.mp4
```

表示从 10 秒处开始，截取 20 秒。  
如果你只做粗剪、并且允许按关键帧切，`-c copy` 很合适。

### 7. 重采样 PCM

```bash
ffmpeg -f s16le -ar 48000 -ac 1 -i in.pcm -f s16le -ar 16000 -ac 1 out.pcm
```

这类命令在语音算法、AEC、ASR 前处理场景中很常见，因为很多算法都要求固定采样率和声道数。

## 7. 哪些场景适合 `-c copy`

经验上，这几类任务优先考虑 `-c copy`：

- 换容器
- 抽流
- 不要求重新压缩的裁剪
- 快速测试媒体链路

因为一旦重编码，你就要面对：

- 编码速度
- 码率选择
- 画质 / 音质损失
- 编码器兼容性

能不转码就别转码，这条经验非常值钱。

## 8. 什么时候必须转码

下面这些场景通常没法只靠拷贝流完成：

- 目标设备不支持原编码
- 需要降低码率或体积
- 需要改分辨率
- 需要改音频采样率或声道数
- 需要做滤镜处理
- 需要生成特定流媒体输出格式

例如：

- H.265 视频要兼容旧设备，可能得转回 H.264
- 48 kHz 双声道音频要给语音模型使用，可能得转成 16 kHz 单声道

## 9. 使用 FFmpeg 时最实用的几个建议

### 先明确目标，不要先写命令

在敲命令前先回答这几个问题：

- 目标平台支持什么容器
- 目标平台支持什么编码
- 是否要求实时
- 是否允许失真
- 是批处理还是单次调试

### 先确认输入，再确认输出

不要“见文件就转”。先 `ffprobe` 看输入，再根据目标输出要求决定：

- 拷贝
- 转码
- 重采样
- 分片

### 参数尽量显式

处理音频时尤其建议明确：

- `-ar`
- `-ac`
- `-b:a`
- `-c:a`

否则输出结果可能依赖默认值，不利于复现和排查。

### 善用帮助系统

很多时候没必要到处搜博客，FFmpeg 自带帮助已经够强：

```bash
ffmpeg -h muxer=flv
ffmpeg -h filter=atempo
ffmpeg -h encoder=libx264
```

## 10. 入门 FFmpeg 的推荐顺序

如果你要把 FFmpeg 学得更稳，建议按下面顺序推进：

1. 容器、流、编码格式、编码器的区别
2. 学会用 `ffprobe` 看媒体信息
3. 学会 `-c copy` 做无损搬运
4. 学会音频转码与重采样
5. 学会裁剪、抽流、简单合并
6. 再进入过滤器、HLS、复杂处理链

## 11. 总结

FFmpeg 真正的门槛不是命令长，而是概念混在一起。只要先想清楚下面四个问题：

- 文件外壳是什么
- 里面有哪些流
- 每条流用了什么编码
- 目标输出到底要什么

大部分命令都会自然变得可推导，而不是只能依赖“复制一条网上命令试试看”。
