Agent 最值得理解的，不是某个框架怎么写，而是它的运行闭环。

## 1. 一个典型 Agent 循环是什么样

大多数 Agent 的流程都可以概括成：

1. 接收目标
2. 判断当前任务状态
3. 决定下一步动作
4. 调用工具执行
5. 观察结果
6. 更新状态并继续

这个循环有时会被解释成类似 ReAct 的思路：  
先思考，再行动，再根据观察结果调整下一步。

## 2. 为什么“反馈”这么关键

很多人只看到了“规划”和“执行”，但实际让 Agent 变得可用的，是反馈。

因为真实世界里经常会发生：

- 搜索结果不够好
- API 调用失败
- 文件格式不符合预期
- 代码运行报错

没有反馈回路，Agent 只会机械执行一次；  
有反馈回路，它才可能修正方向。

## 3. 一个简单案例怎么理解

比如天气助手这种最小例子，流程其实很清楚：

- 从用户输入里提取城市
- 调天气工具
- 拿到结构化结果
- 生成建议
- 把中间结果存起来

这已经包含了最基本的 Agent 雏形。

## 4. 工程上真正需要关注什么

做 Agent，不要只盯“模型够不够强”，还要一起看：

- 状态怎么表示
- 工具接口是否稳定
- 失败怎么处理
- 中间步骤是否可观测

如果这些没理顺，模型再强，系统也很难稳定。
