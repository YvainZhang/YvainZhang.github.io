很多人第一次接触 Agent，都会把它理解成“更聪明一点的聊天机器人”。这不算错，但不够用。

更贴近工程的说法是：Agent 是一个能接收目标、拆分步骤、调用工具并持续推进任务的执行系统。

## 1. 它和普通大模型差在哪

普通大模型更像一次性生成器：

- 你给输入
- 它给输出

而 Agent 更像带状态的执行者：

- 接收目标
- 判断当前要做什么
- 调用工具
- 观察结果
- 决定下一步

所以 Agent 的核心不是“生成更长的回答”，而是“围绕目标持续行动”。

## 2. 为什么现在 Agent 会变得重要

因为很多真实任务本来就不是一句话能结束的，比如：

- 做信息搜集与整理
- 跑一段代码并修正错误
- 读取文档后生成结论
- 在多个工具之间搬运上下文

这些任务都需要过程，不只是结果。

## 3. 一个最简单的 Agent 应该具备什么

最小结构通常就四样：

- 理解输入的能力
- 拆解任务的能力
- 调用工具的能力
- 保存状态的能力

也就是常见的那条公式：

`LLM + Planning + Tool use + Memory`

## 4. 什么时候不需要 Agent

并不是所有问题都适合上 Agent。  
如果你的任务只是：

- 问一个定义
- 写一段固定格式文案
- 做单步分类或抽取

那普通 prompt 往往已经够了。  
只有当任务需要多步推进、状态保留和工具交互时，Agent 的价值才会真正体现出来。
